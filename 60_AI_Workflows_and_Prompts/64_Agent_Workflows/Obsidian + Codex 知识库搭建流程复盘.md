# Obsidian + Codex 知识库搭建流程复盘

## 外部流程参考与本库适配

总体结构流程参考：

- [知乎专栏：知识库结构流程参考](https://zhuanlan.zhihu.com/p/2048182242585924941)

适配原则：

- 外部文章只作为流程骨架参考，不直接照搬目录名称和公司型权限模型。
- 对每一步流程，可让 Gemini 先参考外部文章生成通用版本，再由 Codex 按本人的专业领域、Obsidian vault 结构和规则文件改写成本库版本。
- 公司知识库模板中的 `COMPANY_RULES.md` 在本 vault 中对应 `WORKSPACE_RULES.md`。
- 通用的 `agent.md` 在本 vault 中对应根目录 `AGENTS.md`，用于规定代理启动规则、角色上下文、沟通风格与技术规范。

## 1. 抛出核心模块，生成初始草图

**动作**：明确指定使用的工具链环境（Obsidian + Codex），并一次性列出当前所有跨度较大的核心任务：从偏微分方程推导、医疗 AI 交叉项目，到工程仿真、量化与日常事务。

**目的**：让 AI 快速摸清知识边界和任务深度，先生成一个兼顾理论推导、工程开发与日常管理的初代目录框架。

## 2. 结构解耦与噪音清理，优化层级

**动作**：

- 审视初版框架后，进行精准修剪。
- 将 AI 默认合并的平行领域强制拆分为独立顶级大类，例如不同方向的医疗项目。
- 补充遗漏的基础共享目录。
- 明确剔除 AI 记忆中已经废弃或不相关的历史研究方向。

**目的**：避免文件夹嵌套过深，减少跨领域逻辑干扰，使目录结构聚焦于当前核心主线，并提升 AI 检索上下文的准确率。

## 3. 固化高频工作流，引入标准模板

**动作**：在静态目录搭建完成后，针对最核心、最高频的工作场景，例如“科研与代码复现”，单独建立标准 Markdown 文件结构模板。

**目的**：

- 通过统一输入规范降低个人记录阻力。
- 让 Codex 等 AI 工具在读取笔记时，能够按照固定结构精准抓取关键信息。
- 推荐结构包括：目标、数学基础、代码、实验设置、结果、待解决问题。

## 4. 设立元知识区，完成 AI 系统对齐

**动作**：

- 跳出具体专业业务，单独建立用于管理“AI 工具使用经验”的分类。
- 覆盖技能栈、工作流、提示词库和工具踩坑记录。
- 让 AI 提炼一份全局适用的系统初始化提示词（System Prompt）。

**目的**：将“如何高效使用 AI”本身资产化。通过总结出的 System Prompt，一次性完成 AI 在专业背景、技术栈和排版偏好上的底层身份绑定，为后续高效交互定调。

## 5. AI 建立目录后的标准化步骤

当 AI 已经理解专业领域并生成初始目录后，继续建立以下全局支撑层：

### 0. `AGENTS.md`

**作用**：定义代理启动规则与长期协作人格。

本 vault 中的实际要求：

- 启动后先读 `AGENTS.md`。
- 再读 `WORKSPACE_RULES.md`。
- 读完后暂停，等待用户命令。
- 明确数学推导、代码工程、论文阅读与沟通风格要求。

### 1. 根目录命名

**作用**：确定顶级目录的编号、英文路径与中文语义。

本 vault 的适配方式：

- 使用稳定英文路径，降低跨工具读写风险。
- 使用编号保持排序，例如 `00_`, `10_`, `20_`, `99_`。
- 中文解释放入索引文件与说明文件，不直接依赖中文路径完成组织。

### 2. 全局规则文件

**作用**：把命名规范、目录权责、标签体系、Codex 处理边界写成可被代理反复读取的规则。

本 vault 中对应：

- 根目录 `WORKSPACE_RULES.md`
- 文件命名规范：`YYYYMMDD - [学科大类/项目] - 核心概念/论文简写.md`
- 唯一入口：`00_Inbox_and_Journal/02_Inbox`
- 正式目录迁移必须遵守标签、摘要、双链与人工确认规则。

### 3. 全局标签页

**作用**：把标签体系从规则文字变成可维护、可扩展、可检索的标签索引。

建议沉淀内容：

- 知识维度：`#Math/FEM`, `#Medical_AI/CBCT`, `#Engineering/Simulink`
- 状态维度：`#Status/To_Read`, `#Status/Proving`, `#Status/Sedimented`
- 工具与数据实体：`#Tool/Codex`, `#Dataset/WestChina`
- 每个标签对应的使用边界、适用目录与典型文档类型。

## 参考 Prompt：从公司知识库规则改写个人知识库规则

以下 prompt 不是本 vault 的生效规则，而是“让 AI 参考并改写规则文件”的模板来源。使用时应要求 AI 将公司型目录、权限与日期格式改写为个人知识库版本。

```text
参考以下的命名风格和规范，给我的

`2、全局规则文件COMPANY_RULES.md`:

# 通用软件公司知识库 使用规范 & Codex 处理规则

## 一、文件命名规范（强制统一）

格式：`YYYY-MM-DD - [部门/项目/分类] - 标题.md`

示例：
2026-06-08 - Tech - 微服务架构设计.md
2026-06-08 - PreSales - XX客户项目方案.md
2026-06-08 - Project - XX系统定制开发周报.md

## 二、目录使用权责

1. **00_Inbox**：所有临时内容、沟通记录、需求草稿、BUG记录、外部资料，必须先存入此处，禁止直接写入正式目录。
2. **01_Company**：公司资质、制度、宣传材料，全员可读，仅行政/管理员编辑。
3. **02_Departments**：各部门独立知识资产，跨部门仅查阅，无修改权限。
4. **03_Projects**：按单个项目建独立文件夹，存放立项、需求、开发、测试、实施、验收全生命周期资料，项目成员专属维护。
5. **04_General_Library**：全公司可复用资产，包括产品资料、技术组件、行业案例、标准合同，鼓励全员沉淀、复用。
6. **05_Templates**：公司统一模板，对外输出、内部流程文档必须套用。
7. **06_Archive**：完结项目、过期文档、废弃方案统一归档，仅查询不日常编辑。
8. **07_Admin**：脚本、日志、标签规范，由知识库管理员维护。

## 三、标签体系规则

统一格式：`#分类-模块-文档类型`，所有文档必须打标签。

## 四、Codex AI 强制处理规则

1. 执行操作前优先阅读本规则，严格遵循目录、命名、标签规范。
2. 区分**标准化产品、定制项目、外包项目**三大类型，精准分类归档。
3. 自动生成80~120字摘要，为关联文档建立双向链接，方便溯源查阅。
4. 客户信息、商务报价、核心源码密钥、涉密需求自动脱敏处理。
5. 重复文档、冗余内容自动合并，空文件、无效链接定期清理。
6. 可复用的方案、代码逻辑、流程文档，主动归集到 `04_General_Library`。
7. 无法判定归属的内容，保留在 00_Inbox 并标注【待人工审核】。
8. 项目结项后，自动梳理可复用资产，再迁移至归档目录。

## 五、安全与协作规范

1. 核心技术架构、未公开产品规划、客户涉密资料，严格限制访问权限。
2. 全库基于私有 Git 做版本管理，所有修改留痕，支持回溯。
3. 每周执行全库备份，日志统一存入 07_Admin/Backup_Log。
4. 版本迭代、需求变更、BUG 修复全程记录，保证项目可追溯。
```

## 可复用方法

1. **先全量暴露任务谱系**：不要过早纠结目录名，先让 AI 看见真实任务跨度。
2. **再做结构裁剪**：对 AI 生成的初版框架进行拆分、合并、删除与补全。
3. **参考外部流程，改写为自己的版本**：让 AI 参考通用知识库搭建流程，但必须按本人的研究方向和工具链重写。
4. **补齐全局支撑层**：建立 `AGENTS.md`、`WORKSPACE_RULES.md`、根目录命名规范和全局标签页。
5. **用模板固定高频流程**：把“常做的事”变成标准输入格式。
6. **沉淀元知识**：把提示词、工具配置、工作流经验和系统指令都纳入知识库。

## 适用场景

- 初始化新的 Obsidian vault。
- 重构跨学科研究知识库。
- 为 Codex / Gemini / Claude / OpenClaw 等 AI 工具建立长期上下文。
- 将项目目录从“资料堆积”升级为“可被 AI 检索和协作的知识系统”。

## 6. 自动化边界确认流程：从规则到 B3 半自动治理

在给仓库添加 Codex 自动化前，先进行边界确认，而不是直接让 AI 写脚本或定时任务。

### 核心提示词

```text
现在我会给这个仓库添加规则和 Codex 自动化，我们先讨论清楚再动手。
拟定流程如下：
1. Capture：所有非结构化信息进入 Inbox。
2. Organize：AI 代理定时清洗、分流、补标签、建双链。
3. Distill：子课题闭环后萃取理论、代码、Prompt 和方法论资产。
4. Retrieve：新任务启动时基于全库上下文检索。
```

### 关键问答路径

1. **权限模式确认**
   - 问：整理 Inbox 时采用建议模式、半自动模式、还是全自动模式？
   - 答：选择半自动模式，即低风险自动移动，高风险等待确认。

2. **风险边界确认**
   - 问：哪些内容必须人工确认？
   - 初版边界偏保守，随后要求“再放宽点”。
   - 最终边界：不涉及理论正确性、医学安全、真实控制决策的内容可自动整理；数学定论、TCI 安全边界、核工程核心控制、医学隐私和新主题首次迁移必须等待确认。

3. **实现路径确认**
   - 初步方案包括 B1 手动触发、B2 脚本辅助、B3 定时自动化。
   - 最终选择直接进入 B3：定时自动化 + 周期性维护报告。

4. **运行频率确认**
   - Inbox Triage：每 4 天 `18:00`
   - Vault Maintenance：每两周周一 `18:15`
   - 时区：`Asia/Shanghai`

5. **报告落点确认**
   - 整理报告追加到 `00_Inbox_and_Journal/01_Daily_Notes/YYYY-MM-DD.md`
   - 双周维护报告写入 `00_Inbox_and_Journal/01_Daily_Notes/YYYY-MM-DD - Biweekly Vault Maintenance.md`

6. **设计分段确认**
   - 自动化架构与数据流。
   - 风险分级与自动处理策略。
   - 文件命名、YAML、正文结构与附件规则。
   - 目录分流规则。
   - 日报与周报格式。
   - Distill 与 Retrieve 规则。
   - 自动化边界、失败处理和审计日志。

### 可复用经验

- 先确认“AI 有多大权限”，再讨论“AI 怎么做”。
- 高风险边界不要只按目录划分，要按内容语义划分。
- 自动化语言要限权：允许“草稿、初筛、待验证”，禁止“已证明、最终结论、临床建议、控制策略”。
- 对科研 vault 来说，自动化的第一目标不是省掉人工判断，而是把人工判断集中到真正高风险的节点上。
- 所有自动化都要有报告和审计日志，保证可追溯。

## 7. 可复用初始化手册：Step + Prompt + Skill

目标：让其他人可以用“网页端有记忆的 Gemini/GPT + 初始化 vault 的 Codex agent + Superpowers brainstorming”复现一套可运行的 Obsidian 知识库。

### 使用角色约定

- **网页端有记忆的 Gemini/GPT**：负责长期背景抽取、专业领域画像、目录草图、规则初稿。适合使用已有记忆和跨轮对话经验。
- **初始化 vault 的 Codex agent**：负责读取本地目录、创建文件夹、写入规则文件、配置 MCP、落地自动化。适合执行本地文件操作。
- **Superpowers brainstorming**：用于任何规则、自动化、目录结构或工作流设计变更前的边界确认。先讨论、确认风险和设计，再实施。

### Step 0：准备个人任务谱系

**对象**：网页端有记忆的 Gemini/GPT

**目的**：把个人长期任务、专业方向和项目边界抽取出来。

**Prompt**：

```text
我准备搭建一个长期 Obsidian + Codex 个人知识库。

请基于你对我的长期了解，帮我整理我的知识任务谱系：
1. 核心专业领域；
2. 当前长期研究方向；
3. 工程与项目方向；
4. 日常记录与资料流；
5. AI 工具使用与 Prompt 工作流；
6. 哪些旧方向应排除，避免污染新知识库。

输出要求：
- 不要给目录结构；
- 只输出任务谱系、边界、优先级；
- 标注哪些内容需要高严谨性或人工确认。
```

**产出**：

- 个人知识边界。
- 当前项目清单。
- 应排除的旧方向。
- 高风险知识类型。

### Step 1：生成第一版目录草图

**对象**：网页端有记忆的 Gemini/GPT

**目的**：先得到概念层面的目录结构，不直接落盘。

**Prompt**：

```text
基于刚才整理的任务谱系，请帮我设计一个 Obsidian vault 初始目录结构。

要求：
- 不套用通用 PARA 模板；
- 顶级目录控制在 6 到 9 个；
- 不同学科主线要解耦；
- 必须包含 Inbox、日记、模板、附件、AI 工作流区；
- 每个目录说明用途、收纳边界和不应放入的内容；
- 输出为树状结构 + 每个目录的说明。
```

**确认点**：

- 是否有不该合并的平行领域被合并。
- 是否有已经废弃的方向被带入。
- 是否存在嵌套过深。
- 是否缺少 Inbox、模板、附件、AI 工作流区。

### Step 2：结构解耦与降噪

**对象**：网页端有记忆的 Gemini/GPT

**目的**：把第一版目录改成个人专属版本。

**Prompt**：

```text
请根据以下反馈重构目录：
1. 将被合并的平行领域拆成独立顶级类；
2. 删除以下不再使用或不相关的方向：[列出旧方向]；
3. 补充共享基础目录：[列出缺失目录]；
4. 保留英文路径和数字编号，中文解释放在说明中；
5. 输出最终建议目录，不要创建文件。

反馈如下：
[粘贴你的修改意见]
```

**产出**：

- 可交给 Codex agent 落盘的目录结构。
- 每个目录的用途说明。

### Step 3：让 Codex 初始化 vault

**对象**：初始化 vault 的 Codex agent

**目的**：把目录、索引、模板真正写入本地 Obsidian vault。

**Prompt**：

```text
请初始化当前 Obsidian vault。

先检查当前目录和 Obsidian MCP 是否可用，然后按下面目录结构创建文件夹、索引文件和基础模板。

要求：
- 保留已有文件，不移动或删除旧内容；
- 创建 Home.md 作为入口；
- 每个顶级目录和二级目录都创建一个简短索引说明；
- 创建 99_Meta/Templates，并加入 Daily Note、Meeting Note、Research Reproduction、Prompt 模板；
- 如果 MCP 不可用，先用本地文件系统初始化，并报告 MCP 状态。

目录结构：
[粘贴最终目录结构]
```

**产出**：

- 本地目录骨架。
- `Home.md`。
- 每个目录的索引文件。
- 基础模板。
- MCP 可用性检查结果。

### Step 4：建立 `AGENTS.md`

**对象**：初始化 vault 的 Codex agent

**目的**：写入代理启动规则、角色上下文、沟通风格和技术规范。

**Prompt**：

```text
请在 vault 根目录创建 AGENTS.md。

规则：
1. 任何 agent 启动后必须先读 AGENTS.md；
2. 再读 WORKSPACE_RULES.md；
3. 读完后暂停，等待我的命令；
4. 例外：明确命名的 Codex cron 自动化可以在读取规则后继续执行，但只能在自动化 prompt 和 WORKSPACE_RULES.md 边界内行动。

请把以下内容写入：
- 我的专业背景；
- 沟通风格；
- 数学推导规范；
- 代码工程规范；
- 论文阅读规范。

内容如下：
[粘贴个人系统指令]
```

**产出**：

- 根目录 `AGENTS.md`。

### Step 5：建立 `WORKSPACE_RULES.md`

**对象**：网页端有记忆的 Gemini/GPT 先起草，初始化 vault 的 Codex agent 再落盘

**目的**：把知识库运行规则写成代理可读的全局规范。

**给 Gemini/GPT 的 Prompt**：

```text
请参考下面的公司知识库规则风格，把它改写为我的个人 Obsidian + Codex 知识库规则。

要求：
- 不保留公司权限模型；
- 改成个人科研、医疗 AI、工程仿真、AI 工作流场景；
- 明确文件命名规范；
- 明确 Inbox 是唯一入口；
- 明确目录流转规则；
- 明确 Codex 处理边界；
- 明确高风险内容必须人工确认。

参考规则：
[粘贴 COMPANY_RULES.md 风格模板]
```

**给 Codex agent 的 Prompt**：

```text
请把下面这份规则写入 vault 根目录 WORKSPACE_RULES.md。

要求：
- 如已有 WORKSPACE_RULES.md，先读取并合并，不要覆盖有用内容；
- 保持与现有目录名称一致；
- 如果发现目录名、标签体系或 AGENTS.md 有冲突，先暂停并报告冲突；
- 无冲突再写入。

规则内容：
[粘贴 Gemini/GPT 生成的个人规则]
```

**产出**：

- 根目录 `WORKSPACE_RULES.md`。

### Step 6：建立全局标签页

**对象**：网页端有记忆的 Gemini/GPT 先起草，初始化 vault 的 Codex agent 再落盘

**目的**：把标签体系从规则文字变成可维护索引。

**给 Gemini/GPT 的 Prompt**：

```text
请为我的 Obsidian + Codex 知识库设计统一标签体系。

要求：
- 一级领域标签用英文，保证跨工具稳定；
- 研究模块标签可用中文或通用术语；
- 状态、工具、数据集使用嵌套标签；
- 每个正式文档至少包含：1 个领域标签、1 个模块或文档类型标签、1 个状态标签；
- 给出组合使用示例。

我的核心领域：
[粘贴任务谱系]
```

**给 Codex agent 的 Prompt**：

```text
请把下面的统一标签清单写入 99_Meta/Global Tag List.md，并更新：
1. WORKSPACE_RULES.md 中的标签规则；
2. 99_Meta/99_Meta.md 的索引链接。

如果与现有标签规则冲突，先提出建议并暂停；如果只是重复，请合并。

标签清单：
[粘贴标签规则]
```

**产出**：

- `99_Meta/Global Tag List.md`
- `WORKSPACE_RULES.md` 标签规则引用。
- `99_Meta/99_Meta.md` 索引链接。

### Step 7：使用 Superpowers brainstorming 设计自动化

**对象**：初始化 vault 的 Codex agent，显式要求使用 Superpowers brainstorming

**目的**：在创建自动化前确认权限、风险边界、报告格式和失败处理。

**Prompt**：

```text
[@superpowers](plugin://superpowers@openai-api-curated) 使用 brainstorm。

现在我要给这个 vault 添加 Codex 自动化。我们先讨论清楚再动手。

拟定流程：
1. Capture：所有非结构化信息进入 Inbox；
2. Organize：AI 代理定时清洗、分流、补标签、建双链；
3. Distill：子课题闭环后萃取理论、代码、Prompt 和方法论资产；
4. Retrieve：新任务启动时基于全库上下文检索。

请按 brainstorming 流程：
- 先读取当前 vault 规则和目录；
- 一次只问一个关键问题；
- 先确认权限模式；
- 再确认高/低风险边界；
- 再确认运行频率、报告落点、失败处理；
- 在我确认设计前，不要写文件或创建自动化。
```

**关键问答顺序**：

1. 权限模式：建议 / 半自动 / 全自动。
2. 风险边界：高风险必须人工确认，低风险可自动整理。
3. 自动化路径：手动触发 / 脚本辅助 / 定时自动化。
4. 运行时间：每 4 天整理、每两周维护。
5. 报告位置：日报、周报、审计日志。
6. 失败处理：读取失败、分类不足、标签冲突、重名冲突。

**产出**：

- 自动化设计 spec。
- 风险边界。
- 报告格式。
- 实施前的用户确认。

### Step 8：创建 Codex 定时自动化

**对象**：初始化 vault 的 Codex agent

**目的**：把已确认的设计落地为 Codex cron 自动化。

**Prompt**：

```text
跳出 Superpowers 流程，直接按已确认的设计实施。

请创建两个 Codex cron 自动化：
1. Inbox Triage：每 [N] 天 [时间] 扫描 Inbox；
2. Biweekly Vault Maintenance：每 [N] 周 [星期] [时间] 扫描全库维护。

要求：
- 自动化 prompt 必须要求先读 AGENTS.md、WORKSPACE_RULES.md、Global Tag List 和自动化设计 spec；
- 低风险内容可自动整理；
- 高风险内容只生成待确认建议；
- 不删除文件；
- 不覆盖原始内容；
- 不移动附件；
- 不生成临床建议、最终数学结论或真实控制策略；
- 写入日报、周报和审计日志。

实施完成后，验证自动化状态、RRULE、工作目录和 prompt。
```

**产出**：

- 两个 ACTIVE cron 自动化。
- `WORKSPACE_RULES.md` 中的自动化规则。
- `99_Meta/Automation Logs`。
- 自动化设计 spec。

### Step 9：沉淀初始化经验

**对象**：初始化 vault 的 Codex agent

**目的**：把搭建过程本身资产化，方便复制给其他人。

**Prompt**：

```text
请把这次初始化 vault 的经验沉淀到 60_AI_Workflows_and_Prompts/64_Agent_Workflows。

要求：

- 记录每一步使用的是网页端有记忆的 Gemini/GPT，还是本地 Codex agent；
- 记录关键 prompt；
- 记录使用 Superpowers brainstorming 确认边界的问答流程；
- 不要重复创建新文档，如果已有“知识库搭建流程复盘”，就合并进去。
```

**产出**：

- 一篇可复用流程复盘。
- 可复制的 Step + Prompt + Skill 手册。
